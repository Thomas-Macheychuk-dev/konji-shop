## Accepted exception

The product below has no usable image:

```text
External ID: 1170531
Name: Pilnik do paznokci
Required status: draft
```

The original Scene7 asset and every generated fallback returned HTTP 403. Re-running the scraper or image repair does not resolve this source-side failure. Supply a manual image or exclude the product before publication.

Unresolved gallery URLs for other products are not release blockers when the product already has at least one usable image. Magento placeholder URLs are intentionally rejected.

## Delivered pipeline

The complete Medi pipeline consists of:

```text
medi:categories
medi:product-links
medi:crawl-product-data
medi:import
medi:full-catalogue
medi:repair-images
```

The pipeline provides:

- recursive category discovery;
- paginated Magento product-link discovery;
- configurable-product and child-variant extraction;
- exact source product IDs, variant IDs, SKUs, prices and availability;
- idempotent product and variant imports;
- draft-by-default status;
- category, attribute and option synchronization;
- medical-device VAT handling;
- resumable batch execution with atomic manifests;
- retained product-data JSON for database recovery;
- structured JSONL operational logs;
- image quality validation and Scene7 fallback repair;
- rejection of Magento placeholders and undersized image responses.

## Stored operational evidence

Default files generated by a complete run:

```text
storage/app/scrapers/medi/categories.json
storage/app/scrapers/medi/product-links.json
storage/app/scrapers/medi/runtime/manifest.json
storage/app/scrapers/medi/runtime/product-data/batch-*.json
storage/app/scrapers/medi/image-repair-report.json
storage/logs/medi/full-catalogue-<run-id>.jsonl
```

Do not delete the product-data batches immediately after a successful execution. They allow the catalogue database to be rebuilt without recrawling Medi.

Operational JSON and logs are runtime evidence, not source-controlled application files.

## Test database safety

Automated tests must never use the application MySQL database.

The test bootstrap enforces:

```text
APP_ENV=testing
DB_CONNECTION=sqlite
DB_DATABASE=:memory:
CACHE_STORE=array
QUEUE_CONNECTION=sync
SESSION_DRIVER=array
MAIL_MAILER=array
```

It also redirects Laravel cache files away from `bootstrap/cache` so a cached development MySQL configuration cannot override the test environment.

`Tests\TestCase` verifies Laravel's resolved connection before any feature test can run. The suite aborts unless the resolved database is SQLite `:memory:`.

Run tests with Pest:

```bash
docker compose exec app ./vendor/bin/pest
```

Expected verified result:

```text
Tests: 590 passed (4594 assertions)
```

After changes to test bootstrap, PHPUnit configuration, Docker environment variables or Laravel cache handling, verify that the development catalogue still contains 134 Medi products after a focused test before running the full suite.

## Full catalogue refresh: local or staging

### 1. Discover categories

```bash
docker compose exec -T app php artisan medi:categories \
  --save=scrapers/medi/categories.json \
  --request-delay-ms=500 \
  --show-failures
```

### 2. Discover product links

```bash
docker compose exec -T app php artisan medi:product-links \
  --categories-from=scrapers/medi/categories.json \
  --save=scrapers/medi/product-links.json \
  --request-delay-ms=500 \
  --show-failures
```

Inspect the number of product URLs and failures before importing.

### 3. Run a controlled first batch

Use a new runtime directory for each independent catalogue execution:

```bash
docker compose exec -T app php artisan medi:full-catalogue \
  --from=scrapers/medi/product-links.json \
  --runtime-dir=scrapers/medi/runtime \
  --batch-size=5 \
  --max-batches=1 \
  --status=draft \
  --image-limit=5 \
  --timeout=120 \
  --attempts=5 \
  --request-delay-ms=1000 \
  --retry-delay-ms=5000 \
  --show-failures \
  --reset
```

Review the manifest and imported records before continuing.

### 4. Resume the remaining catalogue

```bash
docker compose exec -T app php artisan medi:full-catalogue \
  --from=scrapers/medi/product-links.json \
  --runtime-dir=scrapers/medi/runtime \
  --batch-size=5 \
  --status=draft \
  --image-limit=5 \
  --timeout=120 \
  --attempts=5 \
  --request-delay-ms=1000 \
  --retry-delay-ms=5000 \
  --show-failures \
  --resume
```

Do not add `--reset` when resuming. Keep `--batch-size`, `--status`, image mode and `--image-limit` unchanged for the lifetime of a manifest.

Timeout, attempt and delay values may be increased during a retry.

### 5. Repair and audit images

```bash
docker compose exec -T app php artisan medi:repair-images \
  --runtime-dir=scrapers/medi/runtime \
  --image-limit=5 \
  --report=scrapers/medi/image-repair-report.json \
  --show-failures
```

The command may return exit status `1` when a product still has no usable image. Review the report rather than assuming the entire import failed.

The accepted final exception is external ID `1170531` only.

## Retry after a transient crawl failure

A failed batch retains its starting offset. Do not reset the runtime.

Resume with increased network limits:

```bash
docker compose exec -T app php artisan medi:full-catalogue \
  --from=scrapers/medi/product-links.json \
  --runtime-dir=scrapers/medi/runtime \
  --batch-size=5 \
  --max-batches=1 \
  --status=draft \
  --image-limit=5 \
  --timeout=120 \
  --attempts=5 \
  --request-delay-ms=1000 \
  --retry-delay-ms=5000 \
  --show-failures \
  --resume
```

Inspect the manifest after the batch advances, then resume without `--max-batches`.

## Database recovery from saved batches

Use this procedure when application data was cleared but the completed runtime product-data files remain available.

It does not recrawl Medi.

```bash
cd /var/www/konji-shop
mkdir -p storage/logs/medi-database-restore
set -o pipefail

mapfile -t BATCHES < <(
  find \
    storage/app/scrapers/medi/runtime/product-data \
    -type f \
    -name 'batch-*.json' \
    | sort
)

if [ "${#BATCHES[@]}" -eq 0 ]; then
  echo "No saved Medi product-data batches were found."
  exit 1
fi

for BATCH in "${BATCHES[@]}"; do
  RELATIVE_PATH="${BATCH#storage/app/}"
  LOG_NAME="$(basename "$BATCH" .json).log"

  docker compose exec -T app php artisan medi:import \
    --from="${RELATIVE_PATH}" \
    --status=draft \
    --image-limit=5 \
    --show-failures \
    2>&1 | tee "storage/logs/medi-database-restore/${LOG_NAME}"

  STATUS=${PIPESTATUS[0]}

  if [ "$STATUS" -ne 0 ]; then
    echo "Import failed for ${RELATIVE_PATH}."
    exit "$STATUS"
  fi
done
```

Run `medi:repair-images` after the batch restore.

## Final audit query

Run after every full import, repair or database recovery:

```bash
docker compose exec -T app php <<'PHP'
<?php

require 'vendor/autoload.php';

$app = require 'bootstrap/app.php';
$app->make(Illuminate\Contracts\Console\Kernel::class)->bootstrap();

use App\Models\Product;

$products = Product::query()
    ->where('external_source', 'medi')
    ->withCount([
        'categories',
        'variants',
        'images',
        'variants as default_variants_count' => fn ($query) => $query
            ->where('is_default', true),
    ])
    ->get();

echo 'Medi products: '.$products->count().PHP_EOL;
echo 'Variants: '.$products->sum('variants_count').PHP_EOL;
echo 'Images: '.$products->sum('images_count').PHP_EOL;
echo 'Products without categories: '
    .$products->where('categories_count', 0)->count().PHP_EOL;
echo 'Products without variants: '
    .$products->where('variants_count', 0)->count().PHP_EOL;
echo 'Products without images: '
    .$products->where('images_count', 0)->count().PHP_EOL;
echo 'Products without default variant: '
    .$products->where('default_variants_count', 0)->count().PHP_EOL;
echo 'Products with multiple default variants: '
    .$products->where('default_variants_count', '>', 1)->count().PHP_EOL;

$duplicateProducts = Product::query()
    ->where('external_source', 'medi')
    ->selectRaw('external_id, COUNT(*) AS total')
    ->groupBy('external_id')
    ->havingRaw('COUNT(*) > 1')
    ->count();

echo 'Duplicate product external IDs: '.$duplicateProducts.PHP_EOL;

$duplicateVariants = DB::table('product_variants')
    ->join('products', 'products.id', '=', 'product_variants.product_id')
    ->where('products.external_source', 'medi')
    ->whereNotNull('product_variants.external_variant_id')
    ->selectRaw(
        'product_variants.product_id, product_variants.external_variant_id'
    )
    ->groupBy(
        'product_variants.product_id',
        'product_variants.external_variant_id'
    )
    ->havingRaw('COUNT(*) > 1')
    ->get()
    ->count();

echo 'Duplicate variant external IDs: '.$duplicateVariants.PHP_EOL;

foreach ($products->where('images_count', 0) as $product) {
    echo implode(' | ', [
        "id={$product->id}",
        "external_id={$product->external_id}",
        $product->name,
    ]).PHP_EOL;
}
PHP
```

## Acceptance checklist

A Medi execution is acceptable when:

```text
runtime status is completed
next_offset equals total_product_urls
crawl_failures is 0
import_failures is 0
all imported products are draft unless explicitly approved otherwise
no product lacks categories
no product lacks variants
exactly one default variant exists per product
there are no duplicate external product IDs
there are no duplicate external variant IDs per product
all products except documented upstream exceptions have a usable image
automated tests pass on isolated SQLite without changing MySQL data
```

Current acceptance: **approved with one documented missing-image exception**.
